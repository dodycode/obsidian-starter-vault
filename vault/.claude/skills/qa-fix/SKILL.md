## Role boundary — read first

This skill runs in the **worktree (implementation) session**. Browser-level retest is **NOT** in scope here — it belongs to the Dev Control orchestrator (the vault session),.

| Step | Owner |
|---|---|
| Read QA `[-]` items + recorded root-cause hints | `/qa-fix` (worktree) |
| Plan + execute the code fix | `/qa-fix` (worktree) |
| Run typecheck + lint | `/qa-fix` (worktree) |
| Stage, commit (on user go-ahead), push | `/qa-fix` (worktree) |
| Hand off context for retest | `/qa-fix` (worktree) — writes "Code-fix landed. Awaiting orchestrator browser retest." into the QA file with the fix-commit SHA, files touched, fix shape, and re-test instructions |
| **Browser retest, screenshot capture, archive moves, QA `[-]`→`[x]` flips, PR Test Plan sync** | **Dev Control orchestrator (vault session) — NOT this skill** |

**Why split this way:** the worktree session is stuffed with diff context + plan-mode reasoning. That's the wrong starting context for browser retesting. A fresh orchestrator session loads the QA file, runs the browser test against the now-deployed fix, captures the pass screenshot, archives the fail, flips the checklist, syncs the PR Test Plan — all without re-polluting the implementation context. Two distinct roles, two distinct sessions. This is the same boundary `/qa-feedback` (orchestrator) ↔ `/qa-fix` (worktree) maintain.

**Hard rule:** if a `/qa-fix` invocation feels like it should reach for Chrome DevTools MCP, `take_screenshot`, or any vault-side QA-file flip — STOP and hand back to the orchestrator instead. The fix-commit SHA + handoff context is what closes the loop on this side.

## What This Skill Does

Reads a QA checklist (generated by `/qa` and updated by `/qa-feedback`), then fixes the `[-]` failed items in this loop:

1. **Locate + parse** the QA file. Each `[-]` item has the verdict + root-cause file:line + screenshot wikilink already inlined (`/qa-feedback` writes them during testing). This skill reads the QA file directly.
2. **Cluster failures by root cause**, not by checklist item. If five `[-]` items all point at the same field-leak in `generic-confirmation.tsx`, that's ONE fix, not five. Use the bug-class clustering from the recall handoff in auto-memory if one exists (`project_qa_{identifier}_handoff.md`) — it's already done by `/qa-feedback` Step 8.
3. **For each cluster, enter plan mode FIRST.** Inside plan mode: research the actual code (read every file the triage pointed at, trace upstream/downstream consumers, web-search the framework / library docs when the fix surface is non-obvious). Do not propose a hacky / quick fix. Write a real plan: what files change, in what order, with what semantics, and how typecheck + lint will validate it.
4. **Open a fresh VS Code window** rooted at the worktree so the user can read the cited code in parallel while reviewing the plan.
5. **Exit plan mode for user review.** The user reviews the plan + code at the same time, gives explicit go-ahead (or asks for changes / a different approach).
6. **Execute the fix** in Claude (this skill's main loop, not a forked agent — the executor needs the parent context).
7. **Run typecheck + lint.** Iterate fix → re-run gates until both pass clean.
8. **Stage the diff.** Present a summary to the user. **NEVER auto-commit.** The user reviews the staged diff and explicitly says "commit" / "ship" / "go" before any commit happens.
9. **Write a handoff note into the QA file** for each fixed cluster: a "Code-fix landed (awaiting orchestrator retest)" block with the fix-commit SHA, files touched, fix shape, and reproducible re-test instructions. **Do NOT flip `[-]` → `[x]`, do NOT capture pass screenshots, do NOT move fail screenshots to Archive — those are the orchestrator's actions on the next round.**

**Why this shape:**
- **Plan mode first, no quick fixes**: every fix is researched, justified, and reviewable BEFORE code lands.
- **VS Code window in parallel**: the user shouldn't have to context-switch between the skill's output and the source code. Open the editor once, reference the same files the plan cites.
- **No auto-commit**: typecheck + lint passing is necessary but not sufficient. The human is the final reviewer.
- **Cluster by root cause, not by checklist item**: avoids 5 separate commits for a single denylist edit. Cleaner history, smaller diff per cluster.
- **No browser retest in worktree**: see "Role boundary" above. Two-session split keeps each session's context tight.

## Output paths

All QA artifacts live under the vault, resolved via `<vault>` from `CLAUDE.local.md`:

- QA file → `<vault>/Tickets/QA/QA-{identifier}.md`
- Failure screenshots → `<vault>/Tickets/QA/QA-{identifier}-fail-{slug}.png`

## Steps

### Step 1: Parse Arguments & Locate QA File

- `$ARGS` is the QA identifier (e.g., `QA-GH-123`, `QA-<repo>-123`, `QA-PR-3266`, or just `GH-123` — accept either).
- If no arguments, ask the user.
- Resolve the QA file path: `<vault>/Tickets/QA/QA-{identifier}.md`
- If the file doesn't exist, try without/with the `QA-` prefix variant. If still missing, surface to the user — don't proceed.
- Also look for the recall handoff at your Claude Code auto-memory dir at `~/.claude/projects/<project-scope>/memory/project_qa_{identifier}_handoff.md`. If it exists, read it — bug-class clustering and deferred-asks are already there.

### Step 2: Parse Failed Items

Extract from the QA file every `- [-]` line and its indented sub-bullets (verdict, root-cause path:lines, notes, screenshot wikilink). Build a list of failures with:
- `task` — the checklist task it belongs to (`Task 2`, `Task 5`, etc.)
- `action` — the bold action text from the original `- [ ]` line
- `verdict` — the `real-bug | not-bug | unsure`
- `rootCause` — file path with line range
- `notes` — the 1-2 sentence explanation
- `screenshot` — wikilink to the failure screenshot

Skip items where the triage returned `not-bug` (those landed as expected behaviour during the QA loop).

### Step 3: Cluster Failures by Root Cause

Group failures by code area / root-cause file. Two failures pointing at the same `path/to/file.ts` are usually one fix. Use the recall handoff's bug-class breakdown if available — it's already done. If not, build the clustering from the failures you parsed: items sharing a `rootCause` file (or the same fix shape in different files) cluster together.

For each cluster, derive:
- `clusterName` — short label (e.g. "banner-id-leak", "generic-card-companyId-denylist")
- `affectedFiles` — set of all files cited across the items in the cluster
- `affectedItems` — list of QA items the fix will close
- `expectedFixShape` — one-sentence intent

### Step 4: Detect Stack Mode

Determine if the failures span multiple PRs or branches:

**If single PR / single branch:**
- Set `STACK_MODE=false`
- All clusters get fixed on the current branch sequentially.

**If 2+ PRs in a Graphite stack:**
- Set `STACK_MODE=true`
- Map each cluster to the PR / branch where the affected files live
- Sort PRs bottom-to-top in the stack (fix the base first, children after)
- Record the current branch to restore at the end

### Step 5: Open VS Code Window for Parallel Review

Before entering plan mode for the first cluster, open a fresh VS Code window rooted at the worktree containing the affected files:

```bash
code --new-window "{REPO_DIR}"
```

Tell the user: "Opened a VS Code window at `{REPO_DIR}` so you can read the cited files alongside the plan. The plan will reference exact files + line ranges — pull them up there while you review."

If `code` isn't on PATH, fall back to `code-insiders` then surface the failure to the user (don't silently skip — the parallel-review workflow is part of the contract).

### Step 6: Process Each Cluster (Sequential Loop)

For each cluster, in stack order (bottom to top):

#### 6a: Enter Plan Mode

Use `EnterPlanMode`. Inside plan mode:

1. **Read every file in `affectedFiles`** at the cited line ranges + a generous surrounding window (at least ±50 lines, more if the function is short and you need the whole thing).
2. **Read every consumer / caller** of the symbols being changed. Use `rg` to find them. Type-system fan-out matters — a denylist change in `generic-confirmation.tsx` ripples to every confirmation card type that uses the generic fallback.
3. **Web-search if the fix surface is non-obvious.** Examples: how does the framework's hidden-fields denylist convention work in similar projects? Is there an idiomatic sanitization helper already in `internal/core/src/utils.ts` (or the project equivalent) that we should reuse instead of writing a one-off? Is the regex/prompt pattern consistent with how the rest of the codebase handles this case? Don't guess — search.
4. **Write the plan** with: (a) what files change with what semantics, (b) why this approach over alternatives (cite the search results / reasoning), (c) the order of edits + why, (d) the explicit verification step (typecheck + lint commands, what the `/qa-feedback` re-test will look like), (e) any user-call-needed scope decisions.

**Hard rules in plan mode:**
- **No quick fixes.** If the fix shape feels like a workaround, find the root cause.
- **No biome-ignore, no `as any`, no `// removed` comments.** Project-style rules apply.
- **No "leave for follow-up" without an explicit `AskUserQuestion`.** Defer-to-later requires user approval.
- **No silent assumptions about behaviour.** If the fix changes semantics in any non-obvious way, list it explicitly in the plan and ask the user before exiting plan mode.

#### 6b: Exit Plan Mode for User Review

Use `ExitPlanMode` and present the plan. **Wait for explicit user approval** — do not auto-execute. The user reviews the plan in the conversation AND the cited files in the VS Code window in parallel.

If the user pushes back on any aspect (different approach, different scope, different file), re-enter plan mode and revise. Repeat until the user explicitly says "go" / "ship" / "execute" / "approved".

#### 6c: Execute the Fix

Apply the edits per the approved plan. Use `Edit` for surgical changes; `Write` only for new files. After each file edit, verify the change by reading the updated region.

**Do NOT change any logic or behaviour beyond what the plan describes.** If you find an unrelated issue mid-edit, surface it to the user — don't bundle it.

After all cluster files are edited, run `git diff` (via `rtk git diff`) to confirm the diff matches the plan.

#### 6d: Run typecheck + lint

Run the project's typecheck and lint commands. Examples:

```bash
pnpm typecheck    # or tsc --noEmit, or whatever your project uses
pnpm check        # or eslint . / biome check . / npm run lint
```

If anything fails:
1. Show the user the failure output and ask via `AskUserQuestion`: **Auto-fix** (Claude addresses each item, then re-run gates), **Push back** (user disagrees with the rule — they reply to the agent), **Stop** (abandon this cluster's fix loop, leave files staged for inspection).
2. Repeat until both gates pass clean (max 2 auto-fix rounds — if still failing on round 3, surface to the user and pause).

#### 6e: (REMOVED — browser retest belongs to the Dev Control orchestrator)

This step used to capture a pass screenshot via Chrome DevTools MCP and flip the QA `[-]` → `[x]`. **It has been removed from this skill** per the role boundary (top of file). The worktree session does not run browser-level QA verification; that's the orchestrator's job on the next round.

What replaces it: write a **handoff note** into the QA file that gives the orchestrator everything it needs to retest after the fix is pushed + redeployed. The note goes inline under the affected `[-]` item — leave the `[-]` AS-IS, do NOT flip to `[x]`. Use this exact shape:

```markdown
- [-] **{original action text from the QA item}**
  - verdict: {existing line — preserve}
  - Root cause: {existing line — preserve}
  - Notes: {existing line — preserve}
  - Screenshot (fail): [[QA-{identifier}-fail-{slug}.png]] {existing — preserve}
  - **Code-fix landed {YYYY-MM-DD} (awaiting orchestrator browser retest):**
    - Cluster: `{clusterName}`
    - Fix-commit SHA: `{shortSHA}` ({"pushed to origin/{branch}" | "staged, awaiting user push"})
    - Files touched: `{relative/path/one.ts}`, `{relative/path/two.ts}`, ...
    - Fix shape: {one-sentence plain-English summary — what changed and why it now works}
    - Re-test recipe (orchestrator runs this on the next round):
      1. {numbered steps, exact URL, exact fixture/data references — copy from the original failure repro if possible}
      2. {expected behaviour after fix}
      3. {what passes vs what would still fail}
    - Deployment dependency: {"none — frontend-only fix, preview deploys on push" | "background-task / job-runner preview rebuild required (mention how to verify deploy state)" | "schema/migration not required" | other}
    - Plan-mode reference: your plan file location (if any)
```

The `Code-fix landed ... awaiting orchestrator browser retest` block is the **only** thing this skill writes into the QA file per fixed cluster. The orchestrator owns the rest:

- Capture the new `*-pass-*.png` via Chrome MCP (after the fix is deployed).
- Move the existing `*-fail-*.png` to `Tickets/QA/Archive/`.
- Flip the `[-]` → `[x]`, append `Screenshot (pass): [[QA-{identifier}-pass-{slug}.png]]` and `Previous failure: [[Archive/QA-{identifier}-fail-{slug}.png]]` indented sub-bullets.
- Sync the PR Test Plan checkbox (see 6e.1).

If a fix has **no browser surface** (pure backend, prompt, helper, internal type rename — nothing to look at in a browser), say so explicitly in the handoff note (`Re-test recipe: backend-only — orchestrator can flip to [x] without a screenshot once {job / task / endpoint} is verified via your logs / observability stack`). The orchestrator still owns the flip.

#### 6e.1: PR Test Plan sync (orchestrator owns this — don't run from worktree)

Per `/qa-feedback` Step 7, the orchestrator regenerates the PR `## Test Plan` section after each round of pass-screenshot capture. **This skill does not run `gh pr edit`.** The worktree's responsibility is limited to:

1. Pushing the fix-commit so the orchestrator's retest hits the right code.
2. Including the fix-commit SHA in the handoff note (Step 6e block above) so the orchestrator can cite it in the Test Plan inline-note translation.

The orchestrator's translation table (preserved here for reference, NOT executed in this skill):

| Source state | Vault QA file (Obsidian) | PR Test Plan (GitHub) — what the orchestrator writes |
|---|---|---|
| Untested | `- [ ]` | `- [ ]` + `*(⏭ pending)*` |
| Tested + passed | `- [x]` | `- [x]` + `*(✅ — {result})*` |
| Tested + failed (open) | `- [-]` | `- [ ]` + `*(❌ — {observed}. Root cause: {path:lines}. [screenshot]({URL}))*` |
| Code-fix landed, awaiting orchestrator retest | `- [-]` (with this skill's handoff block) | `- [ ]` + `*(🛠 — code-fix landed in commit `{shortSHA}`, awaiting browser retest)*` |
| Fixed + retested + verified by orchestrator | `- [x]` (with pass + archived-fail wikilinks) | `- [x]` + `*(✅ — fixed in commit `{shortSHA}`. [pass]({URL}) \| previous [failure]({URL-Archive}))*` |

`[-]` renders as literal "[-]" text on github.com (GFM only supports `[ ]` and `[x]`) — that's why the orchestrator's PR translation collapses `[-]` to `[ ]` with a 🛠 icon. Vault stays `[-]` for Obsidian half-check rendering; the orchestrator handles the cross-rendering when it syncs the PR.

If the QA file covers multiple PRs (Graphite stack), the orchestrator syncs every PR's Test Plan — also out of scope here.

#### 6f: Present Cluster Summary + Stage (NEVER auto-commit)

Show the user:

```
## Cluster: {clusterName} — Ready for Your Review

### Failures addressed (orchestrator will verify after redeploy)
{list of QA items the fix targets — task name, action, root-cause file}

### Plan executed
{the approved plan, in 3-5 bullet points}

### Verification gates
- typecheck: clean
- lint: clean

### Handoff note written to QA file
{which `[-]` items got the "Code-fix landed (awaiting orchestrator browser retest)" block;
the items remain `[-]` — orchestrator flips to `[x]` after retest passes}

### Deployment dependency for orchestrator's retest
{e.g. "frontend-only — fix lives on origin/{branch}; preview rebuilds on push" |
"background-task preview rebuild required — orchestrator should verify the new task version is live before retesting" |
"backend-only — orchestrator can verify via {logs / job runner / observability dataset / etc.} without a screenshot" |
"none"}

### Diff (staged, NOT committed)
{git diff --stat output}

The diff is staged on `{branch}`. Reply "commit" / "ship" when you're ready to commit. I will NOT commit on my own.
**Browser retest is the orchestrator's job — do not run it from this session.**
```

**Stage the files via `git add`. Do NOT call `git commit` under ANY circumstances.** Even if typecheck + lint pass clean. Even if the user previously said "commit per cluster". The user explicitly opts in to each commit, every time.

#### 6g: Wait for User Decision

Three valid user responses:
- **"commit" / "ship" / "go"** — proceed to 6h.
- **"more changes" / "modify"** — re-enter plan mode for the next iteration on this cluster.
- **"reject" / "discard"** — `rtk proxy git restore --staged .` then `rtk proxy git checkout -- .` (after explicit confirmation), abandon this cluster, move to the next.

#### 6h: Commit (only on explicit user go-ahead)

When the user has explicitly approved:

```bash
rtk git commit -m "fix({scope}): {short description per the plan} [{ticket-id}]"
```

Use Conventional Commits format (see the vault's `gh-hygiene.md`). Scope from the affected app/package. Subject ≤ 50 chars. Body only when the "why" isn't obvious from the subject — when needed, explain WHY (which QA item this fix unblocks), not WHAT.

**Bundle commits where it makes sense** — even though this skill commits per-cluster (the user explicitly opts in each time), prefer to bundle small related clusters when the user agrees, to keep the final review surface singular.

#### 6i: Stack Management (if STACK_MODE=true and more clusters remain)

```bash
rtk gt restack
```

Only restack between clusters — do NOT `gt submit` per-cluster. Submitting happens once at the end (Step 8). If restack has conflicts, stop and ask the user.

### Step 7: Move to Next Cluster

Repeat Step 6 for the next cluster. Re-open VS Code at the new worktree if the cluster sits on a different branch.

### Step 8: Restore & Final Summary

1. If STACK_MODE=true, ask the user: "All clusters fixed. Submit the stack via `gt submit --stack` now, or hold for further review?" Do NOT auto-submit.
2. Checkout the original branch (the one the user was on before the skill started).
3. Print the final summary:

```
## QA Fix Complete: {ticket-id} (worktree round done)

| Cluster | Files | Tech-Lead | Handoff note written | Status |
|---------|-------|-----------|----------------------|--------|
| {name} | {N files} | {approved \| K rounds} | yes — {N} `[-]` items | Committed / Pushed / Skipped / Rejected |
| ... | ... | ... | ... | ... |

QA file at `{path}` updated:
- {N} `[-]` items got a "Code-fix landed (awaiting orchestrator browser retest)" handoff block
- 0 `[-]` → `[x]` flips (intentional — that's the orchestrator's call after retest)
- 0 screenshots captured / archived (intentional — orchestrator owns Chrome MCP)

Next: {if STACK_MODE && user said hold} run `gt submit --stack` when ready. {else if pushed} stack already submitted, deployment in progress. {else} push when ready.

Then hand back to the orchestrator (Step 9).
```

### Step 9: Hand back to the orchestrator (vault) session

The QA file at `{vault path}/Tickets/QA/QA-{identifier}.md` now carries this round's fix-landed handoff blocks. **The orchestrator (vault session) owns the next steps:** browser retest after redeploy, screenshot capture, archive moves, `[-]` → `[x]` flips, PR Test Plan sync. This skill does NOT do any of those.

Surface this to the user explicitly:

```
## Hand back to the orchestrator (mandatory — fix loop is not closed yet)

This worktree session has finished its round. The handoff context is
in the QA file under each affected `[-]` item ("Code-fix landed
(awaiting orchestrator browser retest)" block — fix-commit SHA, files
touched, fix shape, re-test recipe, deployment dependency).

To close the loop, switch to the vault (orchestrator) Claude session and:

1. Wait for the deployment dependency to clear (e.g. preview deploy rebuild,
   background-task version rollout — see the handoff block per item).
2. Run `/qa-feedback {identifier}`. The orchestrator will:
   - Re-test the items with handoff blocks (read the recipe under each).
   - Capture pass screenshots via Chrome DevTools MCP.
   - Move fail screenshots to `Tickets/QA/Archive/`.
   - Flip `[-]` → `[x]` and append the pass + archived-fail wikilinks.
   - Sync the PR Test Plan checkbox to `[x]` + ✅ inline note.
3. If a retest fails (deploy not live, fix incomplete, regression), the
   orchestrator triages back into a fresh `[-]` item and the next
   round comes back here.

Do NOT keep the worktree session running for the retest — that
re-pollutes the implementation context with QA-loop noise. Per
the vault `dev-control-handoff.md` rule, the orchestrator role drives
QA loops; the worktree role drives fixes. Two distinct sessions.
```

If the user says "I'll just re-test here" — that's an explicit override; honor it (the user can use Chrome DevTools MCP from any session), but log the override in this session and remind them the next round of fixes should still come back to a fresh worktree session.

## Notes

- **Plan mode is non-negotiable.** Even for one-line fixes. The user wants a research + reasoning trail before code lands.
- **VS Code window opened ONCE per session, at the start.** Subsequent clusters reuse the same window — only re-open if a different worktree is involved.
- **Triage was already done.** `/qa-feedback` recorded the verdict + root-cause hint per failure. Don't re-judge those during the fix; just execute.
- **Never auto-commit.** Repeating because it matters: even when typecheck + lint pass clean, even when the diff is one line, even when the user previously said "commit per cluster" — the user opts in EACH time. This is the explicit user contract for this skill.
- **Web-search is part of plan mode.** If the fix isn't obvious from reading the code alone, search the framework / library docs / GitHub issues / RFCs. Cite the source in the plan. Don't guess.
- **No "follow-up ticket" escapes.** If a finding is in scope, fix it now. If it's genuinely out of scope, ask the user via `AskUserQuestion` and get an explicit defer.
- **Vault path is the default.** Every reference to the QA file location resolves through `<vault>/Tickets/QA/`, the single source of truth.
- **Screenshot archive workflow is the orchestrator's job, not this skill's.** This skill leaves fail screenshots in place and writes a handoff note. The orchestrator captures the new pass, moves the old fail to `Archive/`, and updates the QA file's wikilinks after retesting against the deployed fix. Don't run Chrome DevTools MCP from this skill.
- **Browser retest is the orchestrator's job.** Same reason — the worktree session has the wrong context for QA verification. See "Role boundary" at the top of this file.
- **Recall handoff is your friend.** If `project_qa_{identifier}_handoff.md` exists in auto-memory, the bug-class clustering is already done — read it instead of re-deriving from scratch.
