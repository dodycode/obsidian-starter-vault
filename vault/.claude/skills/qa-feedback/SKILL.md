## What This Skill Does

Interactive feedback loop for logging manual QA results.

**Two roles, one agent (Claude):**
- **Triage.** For each issue the user reports, Claude semantically matches it to the closest checklist item, judges whether it's a real defect or expected behavior, and points at the most likely root-cause file/area.
- **Executor + scribe.** Claude runs the interactive loop, writes the verdict back to the QA file, and (when the user asks) drives the actual fix — directly or via forked agents.

**Requires:** A QA checklist generated by `/qa` in the vault: `<vault>/Tickets/QA/QA-{identifier}.md`.

---

## Output paths

All QA artifacts live under the vault, resolved via `<vault>` from `CLAUDE.local.md`:

- QA file → `<vault>/Tickets/QA/QA-{identifier}.md`
- Failure + pass screenshots → `<vault>/Tickets/QA/QA-{identifier}-{fail|pass}-{slug}.png`
- Stale failure screenshots (after a bug is fixed) → `<vault>/Tickets/QA/Archive/QA-{identifier}-fail-{slug}.png`

## Steps

### Step 1: Parse Arguments (Claude)

- `$ARGS` = identifier, one of:
  - GitHub issue: `owner/repo#123` (matches `[\w-]+/[\w-]+#\d+`) → resolves to `GH-123` (or `<repo>-123` for multi-repo workspaces)
  - GitLab issue: `namespace/project!123` (matches `[\w-]+/[\w-]+!\d+`) → resolves to `GL-123`
  - Direct QA identifier: `PR-3266`
- If no arguments, ask the user for the identifier.

### Step 2: Find the QA File (Claude)

1. Derive project name from cwd:
   - Get directory name from path under the workspace folder
   - Strip any worktree suffix matching `-{branch-name}`
   - Fallback: git repo root directory name
2. Resolve `{REPO_DIR}` to the worktree containing the changed code (the triage needs a real repo path with the diff to reason about root cause).
3. Resolve the QA filename from the identifier:
   - `owner/repo#123` → `QA-GH-123.md` (or `QA-<repo>-123.md`)
   - `namespace/project!123` → `QA-GL-123.md`
   - `PR-3266` → `QA-PR-3266.md`
4. Look for the QA file under the routing-aware directory.
5. If not found, tell the user: "No QA checklist found. Run `/qa {identifier}` first to generate one."
6. Read the QA file contents in full.

### Step 3: Enter Feedback Loop (Claude orchestrates + triages per issue)

Display current status: count of `[ ]` (untested), `[x]` (pass), `[-]` (fail).

Then start the loop using AskUserQuestion:

```
Testing {identifier}. {X} items untested, {Y} passed, {Z} failed.
Describe an issue you found (or select 'Done' to finish):
```

For each issue the user reports:

#### 3a: Collect User Description + Optional Screenshot (Claude)

- Ask: "Describe the issue (free text, no need to match wording)."
- Ask: "Screenshot path? (paste path or skip)"

#### 3b: Triage the Report Inline (Claude)

Claude does the triage directly — no external tool, no shell-out. Steps:

1. **Match the user's description to a checklist item** by semantic intent — not keyword overlap. The user describes user-facing behavior; checklist items describe expected behavior. The bold action text is the anchor. If no item is a clear match, mark as `NEW`.

2. **Read the diff** in `{REPO_DIR}`. Try the project's default branch:
   ```bash
   DEFAULT_BRANCH=$(gh repo view --json defaultBranchRef -q .defaultBranchRef.name 2>/dev/null || echo main)
   git diff $(git merge-base HEAD "$DEFAULT_BRANCH")..HEAD
   ```
   Read the changed files with the most relevance to the user's description.

3. **Judge whether the report is a real defect:**
   - **real-bug** — the described behavior contradicts the checklist item / acceptance criteria, or violates a project rule in the project's `CLAUDE.md` / vault rules.
   - **not-bug** — the behavior is expected (per the checklist or per a rule).
   - **unsure** — needs clarification (rare; only when the description is too vague to judge).

4. **If real-bug**, name the most likely file (path + approximate line range) the bug lives in. Pick the smallest plausible scope. If you genuinely can't tell, write `n/a`.

5. **Hold the verdict in a 4-line block** that Step 3d will reuse:

```
MATCH: {exact bold action text from a checklist item, or NEW}
VERDICT: {real-bug | not-bug | unsure}
ROOT-CAUSE: {path/to/file.ts:42-60 | n/a}
NOTES: {1-2 sentences explaining the verdict}
```

#### 3d: Confirm with User and Update QA File (Claude)

Show the user the parsed verdict using AskUserQuestion:

```
The triage matched your report to:
  > {MATCH or "NEW (no match)"}

Verdict: {VERDICT}
Likely root cause: {ROOT-CAUSE}
Notes: {NOTES}

How should I log this?
- [Confirm — log as-is]
- [Pick a different checklist item]
- [Log as new issue (Additional Feedback)]
- [Skip — don't log]
```

Then update the QA file:

**If MATCH = `<exact bold action text>` and user confirmed:**

Find the matching `- [ ]` or `- [x]` line and replace it with `[-]`, then append indented bullets below:

```markdown
- [-] **Create a company with duplicate business number** → Error message shown
  - Verdict: real-bug
  - Likely root cause: `internal/operations/src/company/create.ts:42-60`
  - Notes: No uniqueness check before insert; duplicate landed silently in PR #3267.
  - User: no error shown, duplicate gets created silently
  - ![](~/Pictures/Screenshots/duplicate-bug.png)
```

**If MATCH = `NEW` or user picked "Log as new issue":**

Append to an `## Additional Feedback` section at the bottom of the file:

```markdown
## Additional Feedback
- [-] **{user's description, trimmed to a single line}**
  - Verdict: {VERDICT}
  - Likely root cause: {ROOT-CAUSE}
  - Notes: {NOTES}
  - User: {full user description}
  - ![](path/to/screenshot.png)
```

**If VERDICT = `not-bug` and user confirmed:**

Don't mark the item failed. Append a "Triage flagged as expected behavior" note to the existing line and leave its checkbox as-is. Tell the user "Looks like expected behavior — leaving as untested. Override?" and respect their choice.

#### 3e: Continue

Show: "Marked '{item}' as {failed | expected behavior}. Next issue? (or 'Done')"

Repeat from 3a until the user selects "Done".

### Step 4: Summary (Claude)

After the loop ends, show:
- Total issues logged this session
- Updated checklist status: `{X} passed, {Y} failed, {Z} remaining`
- File path for reference
- Triage disagreements: count of issues where the triage returned `not-bug` and the user overrode

### Step 5: Mark Passing Items + Archive Resolved Failures (Claude)

This step has TWO sub-flows depending on the source of the pass mark — they MUST be handled differently because re-tests carry historical fail evidence that must be preserved.

#### 5a: Untested → passed (first-time pass)

For items still marked `[ ]` (never tested in this loop): ask "Want to mark any untested items as passed? I can show the remaining ones."

If yes, show untested items in batches and let the user mark them as `[x]` (pass) or skip. No triage needed — this is manual book-keeping. Capture a pass screenshot per `## Driving the browser` if visual confirmation matters; otherwise the chat-reply text logged inline is sufficient.

#### 5b: Previously failed → now passed (RE-TEST after fix) — MANDATORY archive workflow

**Trigger:** the user reports an item passing that is currently marked `[-]` in the QA file. This typically happens after `/qa-fix` lands the fix and the user re-runs `/qa-feedback` to verify, OR during a same-session re-test in a long QA loop.

**This sub-flow is not optional.** Skipping the archive workflow loses regression evidence and clutters the QA folder with stale fail screenshots. Six months from now, when the same surface breaks again, the archived fail screenshot is the only record of what the original bug looked like.

**For every `[-]` → `[x]` flip during re-test, do ALL of:**

1. **Locate the existing fail screenshot.** The current `[-]` line has `Screenshot: [[QA-{IDENTIFIER}-fail-{slug}.png]]` (or similar). Note the exact filename — preserve it on move.
2. **Capture the new pass screenshot** at `Tickets/QA/QA-{IDENTIFIER}-pass-{slug}.png` (mirror the slug of the fail to make the pair searchable). Use the same Chrome DevTools MCP flow as the original test capture.
3. **Move the old fail screenshot** to `Tickets/QA/Archive/QA-{IDENTIFIER}-fail-{slug}.png` (preserve the filename — do NOT rename). Use `git mv` if the vault is a git repo so the file history follows. **Never overwrite the fail file.**
4. **Update the QA-file line.** Replace `- [-]` with `- [x]`, keep the ORIGINAL bold action text + verdict + root-cause + notes intact (regression history), and append two new indented bullets:
   - `Screenshot (pass): [[QA-{IDENTIFIER}-pass-{slug}.png]]`
   - `Previous failure: [[Archive/QA-{IDENTIFIER}-fail-{slug}.png]]`
   - Optionally: `Fix shipped: <commit shortSHA>` if the user can name the commit.
5. **Partial-pass case** (the original failure is solved but the re-test surfaced a NEW different bug at the same surface): mark `[x]` for the original bug class (it's resolved), then either log the new bug as a fresh `## Additional Feedback` item OR add a `New regression` indented bullet under the same line referencing a new fail screenshot. **Don't keep `[-]` for an item whose original failure is solved** — the old failure deserves the archive treatment regardless of whether the surface has new issues.
6. **If the QA file is in a git repo** (this vault is — most projects' QA folders are too): the Step 7 PR-sync prerequisite chain still applies. The vault must be committed + pushed before the PR Test Plan sync runs, so all `Archive/` paths resolve on github.com.

**Why this is in `/qa-feedback` and not `/qa-fix`:** `/qa-fix` runs the fix loop and archives screenshots for items it itself re-verified. But QA loops also re-test items the user fixed manually, items fixed by a teammate, items fixed in a previous session, or items that fix-as-side-effect-of-another-cluster. `/qa-feedback` is the canonical re-test entry point and has to handle the archive workflow uniformly, regardless of which skill landed the fix.

**Hard rules:**

- **NEVER flip `[-]` → `[x]` without the archive move + pass-screenshot capture.** The QA file's `[x]` carries an implicit regression-evidence contract: pass + archived fail wikilinks are part of what `[x]` means in this folder.
- **NEVER `rm` the fail screenshot.** Move to `Archive/`, never delete.
- **If `Tickets/QA/Archive/` doesn't exist yet,** create it (`mkdir -p`) — it's a vault-level convention, not per-ticket. The folder's `CLAUDE.md` (if present) documents the workflow.
- **If the slug doesn't have a clean fail+pass pair** (e.g. one fail screenshot covered multiple `[-]` items, or a `[-]` item never had a screenshot in the first place): mark `[x]` and add a `Re-tested {YYYY-MM-DD}: pass — original failure documented in {whatever evidence exists}` note. Don't fabricate a missing pass screenshot just to fill the wikilink slot.

### Step 6: Hand Off — Don't Trigger /qa-fix From This Session (Claude)

`/qa-feedback` is the QA-orchestrator role: it logs verdicts, syncs the PR Test Plan, archives screenshots, saves the recall handoff to memory. **It does not implement fixes.** Per the vault `dev-control-handoff.md` rule, the implementation step (`/qa-fix`) runs in a SEPARATE Claude session inside the worktree — fresh context, no QA-loop noise.

**Detect whether handoff is needed:**

```bash
# Are we in a worktree already (NOT the vault, NOT the main app repo)?
# A worktree is a sibling of <project-repo> with a branch suffix.
PWD_PATH=$(pwd)
PROJECT_REPO_NAME=$(basename "<project-repo>")
PROJECT_PARENT=$(dirname "<project-repo>")
if echo "$PWD_PATH" | grep -qE "^${PROJECT_PARENT}/${PROJECT_REPO_NAME}-[a-zA-Z0-9-]+$"; then
  IN_WORKTREE=true
else
  IN_WORKTREE=false
fi
```

**If NOT in a worktree (Dev Control / vault session — the common case):**

Tell the user, verbatim:

> Failed items are marked `[-]` with the root-cause hint inline. PR Test Plan synced. Recall handoff saved to memory.
>
> **Hand off to fix:** cd into the worktree where the PR's branch lives and start a fresh Claude session there:
>
> ```
> cd "<absolute path to <project-name>-{branch-name>"
> # then start Claude — it'll auto-load the worktree's CLAUDE.local.md
> ```
>
> Once the fresh session is up, prompt it: `/qa-fix QA-{IDENTIFIER}`. The fresh session has clean context for implementation; this session keeps the QA-orchestrator state. Hard gate inside `/qa-fix`: typecheck + lint must pass clean before staging the diff. No auto-commit.

Resolve the worktree path from the PR branch name. Common pattern: `<worktree-parent>/<project-name>-{branch-name}` (siblings of `vault/` inside `<workspace>`). If unsure, `git worktree list` from the main app repo shows them.

**If already in a worktree (rare — user re-entered the loop after fix work started):**

Tell the user `/qa-fix QA-{IDENTIFIER}` can run in the current session — it's already the coding-agent context.

**Do not invoke `/qa-fix` directly from this skill.** Even when the handoff is "obvious", the user starts the next session manually so the context split stays clean. The recall-handoff memory entry (Step 8) is what bridges the two sessions.

**The loop:** `/qa-feedback` → handoff → `/qa-fix` (worktree session updates the SAME vault QA file: flips `[-]` → `[x]`, archives fail screenshots, adds pass screenshots, syncs PR Test Plan, commits the vault) → user comes BACK to this orchestrator session → re-runs `/qa-feedback {identifier}` to re-test the just-fixed surfaces (or to triage newly-discovered failures) → handoff again if more `[-]` items land. Repeat until QA is clean. The vault QA file is the round-trip state surface; both sessions read + write to it.

**Idempotent re-entry.** When this skill is re-invoked on a QA file that already has fixes from a previous `/qa-fix` round, the items will already be `[x]` with `Re-tested {date}: pass` notes. Do NOT re-test those — only the `[-]` items still need triage. Step 8's recall-handoff memory entry includes a "what NOT to redo" list so the round-trip stays cheap.

### Step 7: Sync the existing Test Plan in the PR description (Claude — every loop, even green ones)

After every meaningful QA-file update, sync the QA results back to the **existing `## Test Plan` section** in the PR description. **NEVER create a parallel "QA status" section** — the PR author already wrote a test plan; you flip its checkboxes inline with results, and append items the QA exercised that the original plan missed.

**Why:** the QA file lives in the vault (gitignored / vault-only). Reviewers, QA owners, product, and on-call can't see it. The PR description's existing `## Test Plan` IS the team-visible surface — they already know to look there. Adding a parallel "QA status" block fragments the source of truth (now there are two places that disagree). Per the vault `issue-tracking.md` rule — keep status on the artifact the team already watches, in the place they already look.

**How to apply:**

1. Resolve the PR(s) the QA file covers. Read the QA file's metadata line (`**Source:** {ID} | PRs: #{N}`). For each PR:
2. Fetch the current PR body via `gh pr view {N} --json body --jq .body > /tmp/pr-{N}-body.md`.
3. Locate the `## Test Plan` section. If it doesn't exist (rare), **stop and ask the user** — don't silently invent one. Asking is cheaper than guessing.
4. **Map every existing Test Plan item to a QA result.** For each `- [ ]` line in the original plan:
   - Find the matching `- [x]` / `- [-]` / `- [ ]` items in the QA file (by semantic intent — the original plan items are usually high-level, the QA file items are more granular). One Test Plan item often maps to multiple QA items.
   - **Use ONLY GitHub-supported task-list states** for the Test Plan checkbox: `- [x]` (rendered as checked) or `- [ ]` (rendered as unchecked). **NEVER `- [-]`** — GitHub Flavored Markdown only recognizes `[ ]` and `[x]` as task-list checkboxes. `[-]` renders as literal text "[-]" on github.com, breaking the visual scan. Confirmed by [GFM spec](https://github.github.com/gfm/) and [GitHub Docs — About tasklists](https://docs.github.com/en/get-started/writing-on-github/working-with-advanced-formatting/about-tasklists). The `[-]` indeterminate state is a [5+-year-old open community request](https://github.com/orgs/community/discussions/19199), not implemented.
   - **GitHub-native convention** (use this exact mapping). Semantic of the checkbox is **"is this resolved / closed?"** — passing tests + fixed bugs are closed; failures are open work and stay unchecked:
     - **Tested + passed** → `- [x] ... description ... *(✅ — observed: {what worked})*` ← checked because the test passed, nothing more to do
     - **Tested + failed** → `- [ ] ... description ... *(❌ — observed: {what broke}. Root cause: {path:lines}. [screenshot]({URL}))*` ← **UNCHECKED** because the bug is open work. NEVER mark a failure `[x]` — at-a-glance scan reads `[x]` as "done/solved", and reviewers will skip the inline note thinking the failure is fixed
     - **Skipped / not tested** → `- [ ] ... description ... *(⏭ skipped — {reason})*` ← unchecked because it's also unresolved (just for a different reason)
     - **Fixed in a follow-up commit (post-/qa-fix)** → flip from `[ ] ❌` to `[x] ✅` with note `*(✅ — fixed in commit `{shortSHA}`. [pass]({URL}) | previous [failure]({URL-Archive}))*`
   - **Why checkbox = "resolved"**: GitHub renders `[x]` as a green checkmark and PRs visually count `[x]/total` as a "completion progress bar". The team scans for unchecked boxes = "still needs attention". Failures need attention. If you mark a bug `[x]`, the reviewer sees ✓ and moves on — the inline `❌` icon is too subtle to override the green-check signal. The leading icon (`✅` / `❌` / `⏭`) tells you **why** the item is in its current state (passed / failed / skipped); the checkbox tells you **whether action is still needed**.
   - Append a one-sentence inline note in italics describing what was actually observed (≤200 chars; link to the QA file for the long version).
**Build the asset base URL once at the start of Step 7** so every screenshot/QA-file link uses the same prefix:

```bash
VAULT_BASE="https://github.com/<your-account>/<vault-repo>/blob/$(git -C '<vault>' branch --show-current)/Tickets/QA"
# Then in the body: [screenshot](${VAULT_BASE}/QA-${IDENTIFIER}-fail-${slug}.png)
```

5. **Add missing surfaces.** If the QA exercised something the original Test Plan didn't list (common — original plans miss surfaces, QA finds them), append the missing items as new bullets under sub-headings:
   - Group by flag state / scenario the original plan used. If the original used `### Flag-OFF` and `### Flag-ON`, add new items under those headings. If the original was a flat list, introduce sub-headings: `### {flag/scenario} — original test plan` (rewritten existing items) and `### {flag/scenario} — additional surfaces tested by QA (not in original plan)` (new items).
   - Each new item gets the same `- [x|-| ]` mark + inline note + screenshot link format.
6. **Add a one-line preamble** at the top of the rewritten Test Plan section pointing at the vault QA file:

   ```markdown
   > **QA pass complete ({YYYY-MM-DD})** — see `Tickets/QA/QA-{IDENTIFIER}.md` in the team vault for the full {N}-item checklist + per-failure root-cause hints + {N} screenshots. Run `/qa-fix QA-{IDENTIFIER}` in Claude Code to address open failures.
   ```

7. **Add a one-line summary line** at the end of the rewritten Test Plan section:

   ```markdown
   **Counts:** {N pass} ✅ / {N fail} ❌ / {N skipped} ⏭. Open failures cluster into **{N} surgical fix areas** ({comma-separated short labels of the bug classes}).
   ```

8. Replace the entire `## Test Plan` section in the PR body with the rewritten one. Keep everything else above + below untouched.
9. Push via `gh pr edit {N} --body "$(cat /tmp/pr-{N}-new-body.md)"` (write to temp file first — long bodies break shell escaping).

**Hard rules:**

- **NEVER create a `<!-- qa-status:start -->` ... `<!-- qa-status:end -->` block or any other parallel "QA status" section.** Update the existing `## Test Plan` in place. The PR author wrote the test plan for a reason; you sync to it, not around it.
- **NEVER delete the original Test Plan items** — even if QA didn't test them. Mark them `- [ ]` with `*(⏭ skipped — {reason})*` so the gap is visible.
- **NEVER use `- [-]` in PR markdown.** GitHub Flavored Markdown only renders `- [ ]` and `- [x]` as checkboxes ([GFM spec](https://github.github.com/gfm/) + [GitHub Docs](https://docs.github.com/en/get-started/writing-on-github/working-with-advanced-formatting/about-tasklists) — the `[-]` indeterminate state is a [5+-year-old open community request](https://github.com/orgs/community/discussions/19199), still not implemented). `- [-]` renders as literal "[-]" text on github.com.
- **NEVER mark a failure as `[x]`** to "preserve the checkbox visual". The team scans `[x]` = "done"; failures with `[x]` look solved. The QA file in the vault uses `[-]` for failures (Obsidian renders it as half-checked), but PR markdown is github.com territory — translate `[-]` → `[ ]` (UNCHECKED — open work) with `❌` icon in the inline note. Only flip to `[x]` when `/qa-fix` lands the actual fix.
- **Use absolute github.com URLs for vault assets, NEVER relative paths.** PR descriptions live on github.com/{app-repo}; the vault is a separate repo. Relative paths like `Tickets/QA/...` or `[[wikilinks]]` resolve to nothing on the PR page — reviewers see broken links. Build the URL: `https://github.com/{vault-owner}/{vault-repo}/blob/{vault-branch}/Tickets/QA/QA-{IDENTIFIER}-{fail|pass}-{slug}.png`. Read the vault branch from `git -C {vault-path} branch --show-current` (don't hardcode `main` — your branch may differ). If the vault repo is private, only reviewers with access see the screenshots inline; for public vaults, anyone with the link does.

- **VAULT MUST BE COMMITTED + PUSHED before syncing the PR.** The github.com URLs only resolve once the vault commit lands on the remote branch. Hard prerequisite chain: (a) write QA file + screenshots locally → (b) `git -C {vault-path} status` to confirm the new artifacts are tracked → (c) `git -C {vault-path} add ... && git -C {vault-path} commit -m '...' && git -C {vault-path} push` (use the `caveman:caveman-commit` skill for the commit message — short subject, scope = `qa`, body only when "why" isn't obvious) → (d) verify the file exists on the remote via `gh api repos/{vault-org}/{vault-repo}/contents/{path}?ref={branch} --jq .sha` → (e) ONLY THEN run `gh pr edit` to push the URL. If you skip (c)-(d), reviewers click the screenshot and get 404. The whole point is broken on first encounter.

- **Vault commit message convention**: use Conventional Commits (see the vault's `gh-hygiene.md` rule), scope `qa`, subject ≤50 chars, body explains the QA pass identifier + counts (`24 ✅ / 7 ❌`) + bug-class summary if you're committing alongside a session that found bugs. Example: `feat(qa): GH-42 QA pass — 7 fail, 17 pass`. Never skip hooks (`--no-verify`); if a hook complains, fix the underlying issue.
- **Never include code, stack traces, or file content beyond `path:lines`.** Per the vault `issue-tracking.md` rule — the PR Test Plan section is read by non-developers (product, QA, stakeholders). Keep it user-facing.
- **For multi-PR QA (e.g. Graphite stack)**, sync the same Test Plan to EVERY PR the QA covers. Each PR is reviewed independently; each needs the status visible.
- **Skip Step 7 entirely if the QA file is brand new + nothing has been tested yet** (all items still `[ ]`). No signal yet.
- **For draft PRs**: still sync. Drafts are visible to reviewers; QA progress on a draft helps the team know when to start reviewing.
- **Idempotent on re-invocation.** Each `/qa-feedback` run regenerates the entire Test Plan section from current QA-file state; old marks are overwritten, not accumulated. The "additional surfaces" sub-section is rebuilt fresh each time.

### Step 8: Save Handoff to Memory (Claude — only when bugs were found)

If at least one item was marked `[-]` during the loop AND the user is wrapping up (chose "Done" with failures present), save a recall handoff to memory so the next session — likely a fresh context running `/qa-fix` — can pick up cleanly without re-running QA.

(Step 7 — PR-body sync — runs every loop. Step 8 only fires when failures landed and the user is closing the session.)

Write to your Claude Code auto-memory directory: `~/.claude/projects/<project-scope>/memory/project_qa_{identifier}_handoff.md` (`<project-scope>` is the per-project memory dir Claude Code derives from the cwd path):

```markdown
---
name: QA {identifier} handoff for /qa-fix
description: Recall context for next session — QA-{identifier}.md ready for /qa-fix, with bug-class breakdown and root-cause hints from the {YYYY-MM-DD} QA pass
type: project
---

QA pass for {brief feature title} (PR #{N}, branch `{branch}`) completed on {YYYY-MM-DD}. QA file at `{absolute path to QA-{identifier}.md}` has every `[-]` failure logged with the triage verdict, root-cause hint, and screenshot wikilink. Worktree: `{absolute path}`.

**Why:** Next session will run `/qa-fix QA-{identifier}` against this file. Recall surface so the new context can pick up the bug-class breakdown without re-running QA.

**How to apply:** When the user opens a fresh session and says "fix the QA findings" or runs `/qa-fix QA-{identifier}`:

1. Read the QA file. Every `[-]` item has the verdict + root-cause file path + screenshot wikilink.
2. **Bug classes** (group findings into N surgical changes, not one mega-PR):
   - Class A — {short description} ({root cause file:lines}). Observed in {tasks/items}.
   - Class B — ...
   (continue for every distinct root cause cluster — the goal is one fix per code-area, not one fix per failed test item)
3. **Test data left on org(s):** {names + IDs of disposable records to optionally clean up after fixes ship}
4. **What NOT to redo:** don't re-run `/qa` (QA file is the source of truth), don't re-test failed items via Chrome until fixes land.
```

Then append a one-line pointer to `~/.claude/projects/<project-scope>/memory/MEMORY.md`:

```markdown
- [QA {identifier} handoff](project_qa_{identifier}_handoff.md) — {one-line hook: bug count + the dominant class}
```

**Skip Step 7 entirely if all items passed** — there's nothing to recall. Memory's not for green QA runs.

**Don't write the bug-class breakdown by hand** — extract it from the `[-]` items in the QA file. Each `[-]` already has the triage's `ROOT-CAUSE: path/to/file.ts:42-60` line; cluster items by file area or by the type of fix they need (e.g. validation tightening, prompt over-bias, etc.). The clustering itself is your judgment call; the data comes from the QA file.

## Notes

- Never modify items already marked as `[-]` (failed) — don't overwrite previous verdict / user feedback. If a re-report hits the same item, append a second indented `- (re-reported)` bullet underneath.
- If the user provides a screenshot path, use it as-is in the markdown image syntax. Don't validate file existence.

## Driving the browser (Chrome DevTools MCP) during the QA loop

When this skill is used together with browser automation (e.g. Chrome DevTools MCP) to drive the test plan, follow these execution rules — they encode lessons from past sessions:

- **Confirmation cards need an explicit click.** When the agent (or any UI flow) renders a confirmation card with `Looks good` / `Yes, add them` / `Approve` / `Make changes` buttons, those cards do NOT auto-resolve. Calling `wait_for(["Created", "Done", ...])` passively will time out because the create/update never starts. ALWAYS take a snapshot, locate the confirmation button by accessible name, and `click` it before continuing the wait. Same for any "approve / cancel" dialog mid-flow.
- **Never assume the agent's next response shape — snapshot the actual page.** Don't `wait_for(["Looks good", "Approve", ...])` expecting a confirmation card just because the test plan says one is "expected". The agent might instead ask a clarifying question (missing email, missing city, missing ABN), redirect to a different flow, or fail silently. Always re-snapshot after every interaction and read the actual UI state — agent text, status text, button labels, follow-up chips — before deciding the next step. If the next step in the test plan no longer matches the page, log what you observed and adapt (or pause and ask the user) — don't blindly retry the planned step against the wrong page.
- **Snapshot before clicking; UIDs change between snapshots.** The accessibility tree re-numbers element UIDs on every snapshot. Don't reuse a UID across two snapshots — re-snapshot, find the button by its label/role, then click. If a `fill` / `click` fails with "did not become interactive", re-snapshot — the page likely navigated.
- **Live progress banners + status text are user-visible surfaces.** Don't only inspect the agent's chat reply. Inspect the bg-task progress card too: the banner that shows the doTask description, the status line below it (e.g. `Deleting customer ld_*...`), and any "Used N tools" expandable details. Internal-ID leaks (`ld_*`, `cmp_*`, etc.) frequently slip through these surfaces even when the chat reply is clean.
- **Mutating tests need a revert plan up front.** Before sending a request that will mutate org-profile or shared data (org name, business hours, settings), tell the user what will change and either (a) get explicit go-ahead, (b) follow with an automatic revert at the end of the task, or (c) cancel via the confirmation card's "Never mind" button to assess intent without persisting. NEVER mutate org-level data silently.
- **Active UI context biases the agent.** Bare phrasings like "Update company name to X" route to the active customer's `businessName` if a customer detail page is open, but to the org-profile if `/agent` root is the context. Run regression-style tests from the canonical context the PR's test plan implies (usually `/agent` root for passthrough tests) — if the test fails, also retest with the alternate context to isolate whether the issue is regex/prompt vs context-bias.
- **Inline triage gets richer signal with both observations.** When you discover a behavior that depends on phrasing (e.g. "MY company name" works, bare "company name" fails), include BOTH transcripts in the triage reasoning — that lets you point at the specific code path that's branching on the difference (regex helper vs system-prompt block).
- **Save screenshots straight into the vault `QA/` folder, NEVER to `/tmp`.** Whenever you capture a screenshot — for `[-]` (failed) items AND for `[x]` (passed) items where visual confirmation is useful — write it directly to `<vault>/Tickets/QA/QA-{IDENTIFIER}-{fail|pass}-{short-slug}.png`. Reference the image in the QA log via Obsidian wikilink: `Screenshot: [[QA-{IDENTIFIER}-{fail|pass}-{short-slug}.png]]` (NOT a `/tmp` path or absolute path).
- **Take pass screenshots too, not just failures.** When a `[x]` item validates a critical visual element (confirmation card subtitle wording, banner text, post-completion chat reply, chip labels), capture it. The pass screenshot becomes the "what good looks like" reference for future regression QA + the diff against the historical fail screenshot when one is later archived. Skip pass screenshots for items where the chat-reply text alone (logged inline in the QA file) is sufficient evidence.
- **Mark the QA doc IN-LINE for every verified item — do not batch.** When Claude is the tester (browser-driving the test plan directly, not just receiving user-reports), every checklist item that just got verified MUST be flipped from `[ ]` to `[x]` (or `[-]` for failures) IMMEDIATELY in the QA file, in the same turn that captures the screenshot. The marking carries an inline rationale (one line: what was observed) plus the screenshot wikilink when one was captured. **Don't defer this to a final batch pass at end-of-loop** — the user (Dody, repeatedly): "you just generated the screenshot but didn't update the qa doc". Skipping inline marks means the QA file ends the loop full of `[ ]`'s even though the work was done, which forces a re-walk to backfill (slow, error-prone) and produces a misleading PR Test Plan sync in Step 7. Treat the screenshot capture and the QA-file flip as a single atomic step. Step 5 ("Mark Passing Items") is a *fallback* for items that genuinely never got tested in the loop, not an excuse to defer normal pass-marking.
- **Inline-mark template** (use exactly this shape — same form as the failure template, just `[x]` and a single-line rationale rather than the full verdict block):

  ```markdown
  - [x] **<original bold action text>** → <original expected result>
    - <one-line rationale: what was observed, what makes this PASS>
    - Screenshot: [[QA-{IDENTIFIER}-pass-{slug}.png]]   <!-- omit if no screenshot was captured -->
  ```

  For deferred / skipped items (couldn't test this session — environment limitation, not a failure), keep the box `[ ]` and append a one-line rationale explaining WHY it's deferred and WHO will verify it. Example: `*(deferred — `resize_page` 390×844 didn't trigger the mobile breakpoint; QA owner to verify on real device.)*`
- **Never overwrite a fail screenshot when the bug is fixed.** When a finding is fixed and re-verified, MOVE the old fail screenshot to `Tickets/QA/Archive/` (preserve filename) and capture a new pass screenshot at `Tickets/QA/QA-{IDENTIFIER}-pass-{slug}.png` (similar slug to the fail). The QA file's `[-]` line gets flipped to `[x]` with a new "Screenshot (pass): [[QA-{IDENTIFIER}-pass-{slug}.png]]" + "Previous failure: [[Archive/QA-{IDENTIFIER}-fail-{slug}.png]]" pair. Why: regression history matters — six months from now, when the same surface breaks again, the archived fail screenshot tells future-you exactly what the old bug looked like.
- Why no `/tmp`: `/tmp` is wiped on reboot — screenshots saved there die before the bug is fixed. The vault is the canonical historical record per the vault's `issue-tracking.md` rule and `<vault>/CLAUDE.md`. Pass `filePath` to `mcp__chrome-devtools__take_screenshot` directly to write to the vault path; don't save to `/tmp` first and move later (extra step, error-prone).
