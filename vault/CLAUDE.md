# Vault — Root Router

> You are a Claude agent reading this because you have entered an Obsidian vault scaffolded from `dodycode/obsidian-starter-vault`. This file tells you **where things live**, **how to search**, and **where to write**. Read it end-to-end before acting.

## What this vault is

A knowledge + orchestration vault for ONE indie project that uses AI-assisted coding. The vault sits inside a workspace folder alongside the app repo and any worktrees:

```
~/Projects/<project>/
└── <project>-workspace/        ← cloned template repo (rename freely)
    ├── README.md, LICENSE, CHANGELOG.md
    ├── vault/                  ← this folder; Dev Control sessions cd here
    ├── <project-name>/         ← your app repo, sibling of vault/
    └── <project-name>-<branch>/ ← worktrees, siblings of vault/
```

It holds: ADRs, runbooks, per-feature local docs, daily dev logs, an Active Work dashboard, per-issue ticket notes that mirror GitHub Issues, and post-merge spec-archive folders.

| System | Owns |
|--------|------|
| **This vault** | ADRs, runbooks, per-feature docs, daily dev logs, Active Work dashboard, per-issue ticket notes + post-merge SDD archives |
| **App repo** | Code + per-app `CLAUDE.md` files |
| **Worktrees** | Active SDD spec files in `<worktree>/context/` (during build) |
| **GitHub Issues** | Tickets (`<repo>#<num>`) — this vault mirrors them under `Tickets/<repo>-<num>/` |

## Where Claude sessions start

| Session type | cwd | What auto-loads |
|---|---|---|
| Dev Control / vault work | `<vault>` | `vault/CLAUDE.md` + `vault/.claude/skills/` + `vault/.claude/rules/` + your `CLAUDE.local.md` |
| Coding session (spec or implementation) | A worktree at `<workspace>/<project-name>-<branch>` | Worktree's `CLAUDE.local.md` (Template A — written by `/new-worktree`), the app repo's own `CLAUDE.md`, the `/qa-fix` skill copied into each worktree |
| App-repo session (rare — direct on default branch) | `<project-repo>` | App repo's own `CLAUDE.md` |
| Workspace root | `<workspace>` | Only your global `~/.claude/CLAUDE.md` — for git ops on the template repo, not actual work |

If `cd <workspace>` and start Claude expecting Dev Control to load — it won't. Claude Code walks UP from cwd looking for `CLAUDE.md`; it doesn't descend into subfolders. `cd vault/` first.

## How to SEARCH (in priority order)

1. **Feature MOC** — if you know the domain, read `Features/<Domain>/MOC.md` first
2. **Nested CLAUDE.md** — each domain folder has its own routing notes
3. **Obsidian native search** — only when interactive inside the Obsidian app
4. **`rg` / `grep`** — for agents, this is the fastest and most reliable way to find content

### Recipes for agents

```bash
# Find all notes mentioning a ticket
rg -l "<repo>-123" <vault>

# Find all ADRs about auth
rg -l -i "auth" <vault>/Engineering/ADRs

# Find runbooks
ls <vault>/Engineering/Runbooks

# Find your last 7 daily logs
ls -t <vault>/People/<your-name>/Daily | head -7

# Find notes with a specific tag
rg -l "#feature/auth" <vault>

# See what was decided about X
rg -B1 -A5 "decided|decision" <vault>/Engineering/ADRs
```

**Rule of thumb:** for structural questions ("where does this live?") use folder CLAUDE.md + MOC. For content questions ("what did we say about X?") use `rg`. Obsidian native search is for humans.

## Conventions

- **Filenames**: `Title-Case-With-Hyphens.md` for docs, `YYYY-MM-DD.md` for daily notes, `<repo>-1234.md` (or `<repo>-1234/notes.md`) for ticket notes, `ADR-0042-short-slug.md` for ADRs
- **Wikilinks**: use them. `[[Features/Auth/MOC]]` over raw paths when linking within-vault
- **Frontmatter**: minimal. Use `status:`, `owner:`, `type:`, `tags:`, `updated:` when they help filtering. Don't invent bespoke keys
- **GitHub Issue IDs**: every ticket note or ticket-related doc includes the issue ref in the filename or H1 title (e.g. `<repo>-123` or `gh-123` when single-repo)
- **Dates**: always absolute (`2026-04-19`), never relative (`yesterday`, `last week`)

### Ownership (`owner:` frontmatter)

Every file of real content carries an `owner:` field. Valid values:

| Value | Meaning |
|-------|---------|
| `<your-name>` | You are responsible for keeping this current |
| `unowned` | Orphan content (e.g. a feature MOC where no one has claimed the domain yet). Lint surfaces these as candidates to claim |

**Defaults by path:**
- `People/<your-name>/*` → `owner: <your-name>` (implicit, always)
- `CLAUDE.md` (any folder) → `owner: <your-name>`
- `README.md`, `Engineering/*/README.md` → `owner: <your-name>`
- `Features/*/MOC.md` → `owner: unowned` until claimed
- `Features/*/<note>.md`, `Features/*/CLAUDE.md` → `owner: <your-name>`
- `Engineering/ADRs/ADR-*.md` → `owner: <your-name>`
- `Engineering/Runbooks/*.md` → `owner: <your-name>`
- `Tickets/*` → `owner: <your-name>`

## Guardrails

- **NEVER** commit `Passwords.md`, credentials, `.env`, or API keys. That's what `.gitignore` is for. If in doubt, don't commit.
- **NEVER** create new top-level folders inside `vault/`. Fit new content into the existing map.
- **NEVER** delete historical daily notes or ticket notes. Move to `Archive/` instead.
- **Daily note logging**: every coding-agent session MUST append to `People/<your-name>/Daily/YYYY-MM-DD.md` — start with an unchecked item, check off and add detail when done.

## Vault-scope skills + rules

Skills + rules under `<vault>/.claude/` auto-load only when Claude Code starts in the vault tree. They do NOT load in app-repo sessions or worktree sessions (with `/qa-fix` as one exception — `/new-worktree` copies it into each worktree's `.claude/skills/`).

### Skills (`<vault>/.claude/skills/`)

| Skill | Purpose |
|---|---|
| `/refresh` | Sync the Active Work note from daily notes + GitHub Issues + worktree state. |
| `/qa` | Generate a manual QA checklist for a GitHub issue. |
| `/qa-feedback` | Log browser-driven QA feedback against a checklist; Claude triages inline + writes fixes. |
| `/qa-fix` | Apply fixes from a QA audit doc inside the worktree (also copied to each new worktree). |
| `/new-worktree` | Create a git worktree + branch from a GitHub issue (`<repo>#<num>`); scaffolds empty SDD spec stubs in `<worktree>/context/`, writes Template-A dual-mode `CLAUDE.local.md`, copies env files + MCP + `/qa-fix`. |
| `/sync-worktree` | Sync skills/rules/env/MCP between main repo and one worktree (either direction). |
| `/sync-worktrees` | Pull latest default branch + restack every child worktree. Reports per-worktree status (clean / skipped-uncommitted / skipped-conflict). |

### Rules (`<vault>/.claude/rules/`)

| Rule | Purpose |
|---|---|
| `dev-control-workflows.md` | Worktree creation, GitHub Issue lifecycle, Active Work procedures, status checks. Holds the Template-A dual-mode worktree CLAUDE.local.md template. |
| `dev-control-spec-driven.md` | Orchestrator-side SDD intake — source detection + parallel read, GitHub Issue stub creation (Path A/B), worktree provisioning with empty stubs, Phase 5b body enrichment after coding agent's spec phase. The actual SDD spec phase (interview, file shapes, sub-tasks) lives in `~/.claude/rules/spec-driven-development.md` and runs in the worktree, not in the orchestrator. |
| `dev-control-daily-notes.md` | Daily-note structure + ownership. |
| `dev-control-handoff.md` | Orchestrator does NOT implement — hand off to a coding-agent session in the worktree. |
| `issue-tracking.md` | No code/file-paths in GitHub Issue bodies; Ready-for-QA bar. |
| `gh-hygiene.md` | GitHub + git conventions: issue title shape, branch naming, Conventional Commits, PR descriptions, hook wiring. |
